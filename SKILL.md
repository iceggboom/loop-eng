# loop-eng — 迭代闭环交付

把一个（通常模糊的）需求，走成一条**每一步都可验证**的闭环。不是一次性写完，而是切成多轮，每轮交付一个能独立验证的小切片。

## 核心三要素（理解这三点比记步骤重要）

- **切片**：每轮 loop 只做一个「能独立验证」的最小增量（如一个页面、一个端点）。不堆半成品。
- **闸门**：用「可执行闸门」界定 done——把「做完了」这个模糊概念变成一条可执行断言，且必须能**先失败**（先红，改完变绿）。守护测试优先；无法自动化时用可执行验收/快照/契约，但同样必须先失败。没有闸门就没有「完成」。
- **回流**：任何阶段发现新事实（实现时撞见隐藏约束、验证时发现遗漏），不要硬推，回到上游对应阶段修正。这是 loop 之所以是 loop。

## 何时用 / 何时不用

- **用**：新功能、复杂 bug、重构——需要严谨闭环、容易跑偏的多步任务。
- **不用**：一行修复、纯问答、读代码理解——直接做，别套流程拖慢。

## 闭环五阶段

```
①框定 → ②建模 → ③建闸 → ④实现 → ⑤验证
  ▲                                  │
  └──── 新事实 / 新需求 / 验证遗漏 ───┘
```

每阶段统一三段式：**触发条件 → 动作 → 出口标准**。回流时精确定位「回到哪个阶段的哪个出口」。

**loop 终止**：所有批次切片走完⑤且全绿、**verifier approved**、无待回流项、`goal` 达成、**且通过一次端到端联调闸门**（启动主链路真跑走通，不只各切片孤立绿）——即整个 loop 结束。

### ① 框定 Frame —— 把模糊敲成明确

- **触发**：需求进入（通常模糊）
- **动作**：能用代码回答的先探索代码库（别拿能查到的事问用户）；一次只问一个决策，每个给出推荐答案 + 理由。必须敲定：**边界**（做什么/不做什么）、**架构**（数据/状态放哪）、**基准**（冲突时以哪边为准）、**范围/批次**（切几片）、**完成定义**（怎么算 done）；**多页面/前端项目，范围必须含「布局/导航壳」切片**（页面间可达性入口，如侧边栏菜单），不能只切页面漏导航——这是跨切片的集成壳，不属于任何单切片，单独漏切则整体不可用
- **出口**：完成定义写成一句可验证标准，登记进 loop-state 的 `goal`
- 辅助 skill：`mattpocock-skills:grilling`、`superpowers:brainstorming`

### ② 建模 Model —— 对照代码找矛盾（按需）

- **触发**：**仅当**需求涉及领域关系/状态流转/术语冲突时；纯前端展示、CRUD、配置类**跳过**，压成一句确认
- **动作**：cross-reference 用户陈述 vs 代码实际行为，有冲突立刻指出（「你说 X，但代码做的是 Y，哪个对？」）；挑战模糊/重载术语；用具体边界场景压测领域关系
- **出口**：无未消化的矛盾，或矛盾已显式记为待回流
- 辅助 skill：`mattpocock-skills:domain-modeling`

### ③ 建闸 Gate —— 先写一条会失败的闸门

- **触发**：完成定义已定
- **动作**：选闸门层级——
  - **强闸门**：单元/集成测试，先红后绿（默认首选）。**无测试基建时**（如前端项目缺组件测试框架），先花一片切片搭最小测试脚手架，不要因缺基建直接降级
  - **弱闸门**：目标行为确实难以纯函数断言时（PDF 产物/图表渲染/外部 IO 时序、**前端页面渲染/框架集成配置**），分两类——
    - **自动化弱闸门**（快照/契约/脚本断言）：能真红。快照=先写「与预期不符」的可失败断言，**不能只建基线**；契约/脚本断言=先写一条会失败的断言
    - **人工验收 checklist**：人按步骤核验，**不能先红**，不得单独当闸门——必须配一条能失败的自动化断言共同守护
  - **运行时闸门**（前端/框架集成/部署配置类切片**必须额外叠加**）：build/tsc/vue-tsc 只是编译闸门，抓不到运行时问题（依赖 ESM 形态、框架默认 baseURL、proxy/环境配置）。必须启动 dev/服务 + 真请求探针——curl 页面返回非空非错误页 HTML、API 经代理返回预期数据字段——断言可先红。**编译绿 ≠ 运行时能跑**
- **出口**：有一条**现在会失败**的闸门（弱闸门同样必须先失败，用可失败断言而非单纯建基线/人工 checklist；前端/集成切片的运行时闸门也必须先红）；红了才进下一步——这证明闸门真的在守这个边界
- 辅助 skill：`mattpocock-skills:tdd`、`superpowers:test-driven-development`

### ④ 实现 Build —— 小步走到绿

- **触发**：闸门已红
- **动作**：切片粒度判定——**一个切片 = 一个能独立跑绿的闸门 + 一个有意义的提交信息**；超过 3 文件或跨 2 个无关模块就拆。小步改到闸门转绿为止
- **出口**：闸门绿；红灯期间不跳下一阶段、不提交

### ⑤ 验证 Verify —— 独立校验 + 全绿 + 记录

- **触发**：闸门转绿
- **动作**：
  1. 跑客观 gate：全量测试、类型检查（tsc / vue-tsc）、构建（build）——必须全绿
  2. **强制独立 verifier**：spawn 一个独立 subagent 作 checker（maker 不得兼任）。只给它 diff + 闸门 + goal，要求它「**找理由拒绝**」——逐项查：diff 是否最小、有无漏边界、闸门是否真守住了这个边界（而非碰巧通过）、有无预期外副作用；**涉及前端/框架集成/部署配置时，verifier 必须启动 dev/服务真跑相关页面/端点，确认非白屏、数据真返回**，不只静态审 diff；**对照 PRD/需求的功能·导航清单逐项核对完整性**（如一级导航是否都有入口、需求点是否都覆盖），不只切片孤立绿——切片绿 ≠ 整体可用
  3. verifier 判定：approved → 进出口；rejected（必附理由）→ 触发回流（见回流引擎：闸门漏 case 回③ / 实现有问题回④ / 理解偏差回①）
  4. 记录分两处：任务内本轮进度/决策（含 verifier 判定与理由）→loop-state，跨会话领域知识/架构决策→memory
  5. 提示「本切片已是一个可独立 review 的提交单元，是否提交？」（提交时机仍遵循「用户要求才提交」）
- **出口**：客观 gate 全绿 **+ verifier approved** + 本轮 loop-state 状态块更新（含 verifier 判定）
- **降级规则**：环境无 subagent 机制时，做一次 role-separated verifier pass（切换视角重审 diff），但必须在 loop-state 披露「非完全独立校验」；**高-risk 切片（认证/权限/数据迁移/计费/部署设置）不得降级**——必须真独立 verifier 或转 needs-human
- 辅助 skill：`superpowers:verification-before-completion`

## 回流引擎（loop 的引擎，操作化）

**回流触发点**（明确出口）：
- **②建模**发现领域模型错误（关系/状态/术语搞错）→ 回 ②建模 重模
- **④实现**撞见隐藏约束（如发现数据已带某字段）→ 回 ①框定 调整边界
- **⑤验证**发现没覆盖的 case → 回 ③建闸 补测试再改；**verifier rejected** → 按理由回 ③（闸门漏 case）/ ④（实现问题）/ ①（理解偏差）
- **需求变更** → 回 ①框定 重新对齐

**回流操作清单（固定 4 步）**：
1. **冻结当前切片**：已写代码/闸门暂存，不删不提交，标记「待回流」
2. **修正上游**：回到对应阶段（①目标 / ②建模 / ③闸门），更新 loop-state 的 goal、模型或闸门描述
3. **重跑或重写**：分两类——**闸门级回流**（⑤发现 case 漏，补断言重跑红绿）；**目标级回流**（①边界/②模型变了，旧闸门作废，重写闸门，冻结的旧代码可能部分作废）
4. **记回流原因**：在 loop-state 记一条回流记录（触发点 / 新事实 / 回到哪），防止下一轮重复踩

冻结而非删除——保留已做工作可复用，也防止「回流」被滥用成「推倒重来」。回流期间不视为「红灯推进」违规——冻结+记原因后是显式阶段回退，非闷头往下做。

## 跨轮状态（防目标漂移）

每轮从①重启时，上轮验收结论、未消化回流项、目标演进若无载体会导致目标漂移。用 `templates/loop-state.md` 作为跨轮记忆锚点，5 字段：`goal` / `当前轮` / `已完成切片` / `待回流项` / `回流记录`。

**使用规则**：
- ①框定出口填 `goal`；每轮开始更新 `当前轮`；⑤验证出口更新 `已完成切片`
- 回流时第 2 步改 `goal`/模型/闸门，第 4 步追加 `回流记录`；**已回流的项从 `待回流项` 移入 `回流记录`，不在待回流项留存**（避免两处重复）
- 默认在对话内按此结构维护；**切片数 >3 或预计跨会话时落盘**到 `.claude/loop-state/<task>.md` 或 memory——防止上下文压缩丢状态（不强制每个任务落盘，避免污染项目）

## 闸门分层 + 逃生通道

**闸门选择规则**：
- 默认强闸门（单元/集成测试先红）
- 切换弱闸门的判定：**目标行为难以纯函数断言**——如视觉渲染、PDF 产物、外部 IO 时序
- 选弱闸门时必须在 loop-state 标注 `闸门类型:弱 + 理由`，防止「图省事降级」
- **前端/框架集成/部署配置类切片必须叠加运行时闸门**：build/tsc 是编译闸门不是运行时闸门，单独不够——这类切片在编译闸门外，必须加「启动 dev/服务 + 真请求探针」的运行时断言（curl 页面非白屏 + API 经代理返数据），可先红

> ⚠️ **反例警示**：某 V1 走完 15 切片闭环——122 单元测试 + build + vue-tsc + 3 轮独立 verifier 全绿，但真启动 dev 后**首页白屏**：`crypto-js` 的 ESM 形态 + 框架 `walRequest` 默认 baseURL `/mock` 两个运行时 bug，编译层与静态审查全没抓到。**单元绿 ≠ 系统能跑，运行时层必须有独立闸门，build/tsc 不替代。**

**逃生通道**——切片卡死无法转绿时的升级路径：
```
卡死信号：同一闸门连续 2 轮没转绿
→ 1. 拆分：把当前切片再切成更小切片，先让子切片转绿
→ 2. 降级：强闸门→弱闸门（理由只能是「目标行为确实难以纯函数断言」，不可因「过不了闸」降级；降级后仍须有可失败断言）
→ 3. 求助：把卡点+已尝试方案抛给用户决策；用户给方案后记入 loop-state，按目标级回流处理（回①/②重对齐）
任一生效即解除卡死，不允许多轮闷头重试
```
呼应「出 bug 先写重现测试再修复」——卡死本质是闸门无法重现通过，逃生是结构化求助而非盲目重试。

## 与子 skill 的边界

- loop-eng 定位为**总纲调度器**，负责跨阶段闭环编排
- 子 skill 分两类调用：
  - **单阶段工具**（`grilling`/`domain-modeling`/`tdd`/`verification-before-completion`）：在对应阶段内调用，天然不跨阶段
  - **闭环工具**（`brainstorming`/`writing-plans`）：**①框定不调 brainstorming，只参考其提问风格**（一调用就会触发它的 spec→plan 完整流程，与 loop-eng 闭环冲突）；若需求复杂到需要正式 spec，直接走 brainstorming 而非 loop-eng
- 冲突仲裁：当子 skill 流程与 loop-eng 阶段出口冲突时，**以 loop-eng 出口标准为准**（呼应①框定的「基准」决策）

## 每轮铁律

1. **切片可独立验证**：每轮产出一个能单独跑通、单独验证的切片，不留「半截功能」。
2. **红灯不推进**：闸门没绿之前，不进下一轮、不提交。含回流期间——冻结记原因后是显式回退，非违规。
3. **决策随做随记**：任务内进度/决策记 loop-state，跨会话领域知识/架构决策记 memory，别堆在脑子里，下一轮会忘。
4. **切片=提交单元**：每轮切片是一个可独立 review 的提交单元；⑤验证出口提示是否提交，提交时机仍遵循「用户要求才提交」。
5. **卡死即走逃生通道**：同一闸门连续 2 轮不绿即触发逃生通道（拆分→降级→求助），不闷头重试。
6. **⑤验证须独立 verifier**：闸门绿后必须经独立 verifier（subagent）approved 才算 done；rejected 即触发回流。无 subagent 时降级 role-separated pass 并披露，高-risk 切片不得降级。
7. **运行时层不缺位**：前端/框架集成/部署配置类切片，⑤客观 gate 必含一次真启动运行时探针（启动 dev/服务 + 真请求断言），build/tsc 不替代；loop 终止前必跑一次端到端联调闸门（启动主链路走通），不只各切片孤立⑤绿。

## 接到需求的第一步（启动清单）

1. 写下「完成长什么样」——一句可验证标准
2. 初始化 loop-state：填 `goal`
3. 进①框定：先探索代码库，再逐个问决策（边界 → 架构 → 基准 → 范围 → 验证）
4. 判断是否需②建模（涉及领域关系/状态流转/术语冲突才进），否则跳过
5. 决策一定，立刻③建闸：选闸门层级，写一条会失败的闸门让它红
6. 切批次，每批走一遍 ①→⑤；每轮更新 loop-state；全部切片全绿且无待回流项即终止
